Migrate authentication from session cookies to JWT

Kind plan Created 2026-05-25 Owner Jordan Ellis Status Draft Read time ~5 min Source canonical-example

In plain terms

  • What this does for the user: Users stay logged in across both web and mobile without the surprise session timeouts they get today.
  • Why it matters: Mobile launch in June needs cross-device auth; the current cookie scheme only works on the desktop browser.
  • What's being asked: Approve the 3-week migration plan and the 1-week dual-running window before retiring the cookie path.

Outcome

By the end of the plan, all clients (web, iOS, Android) authenticate via JWT issued by a single auth service, with refresh tokens stored client-side and revocation managed via a small blocklist. Cookie-based sessions are retired and removed from the codebase.

Approach

Three stages with a deliberate dual-running window between stage 2 and stage 3. The auth service exists today as a Django app and grows new JWT endpoints; clients are migrated one at a time; the cookie middleware is removed last.

Risks

  • Refresh storm: if refresh tokens have a short TTL and the web client retries on 401, we can hit auth-service throughput limits. Mitigation: 7-day refresh window and exponential back-off on the client.
  • Stale cookie sessions during dual-running: a user might end up authenticated on cookie AND JWT simultaneously. Mitigation: cookie-issued requests get a deprecation header; clients prefer JWT.
  • Mobile rollback complexity: if the mobile launch is delayed past July, the dual-running window extends and the cookie removal slips. Mitigation: cookie removal is gated on a green metric, not a calendar date.

Rollback

Each stage rolls back independently. If stage 1 endpoints are broken: feature-flag them off, no client impact. If stage 2 web migration regresses login flow: re-enable cookie middleware (still present), revert client. If stage 3 mobile launches and breaks: mobile clients fall back to cookie-on-WebView until fixed forward. Stage 4 cookie removal is non-trivial to roll back; it gates explicitly on one week of clean auth metrics.
