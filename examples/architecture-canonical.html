In plain terms

  • What this does for the user: Users get a faster, more reliable AI chat because the MCP service becomes simpler to change, test, and extend.

Candidate 1

Wrapper repetition vs declarative tool registration

PM lead: when a teammate adds a new MCP tool today, they copy ~140 lines from another wrapper. Fixing a bug means replaying it across 50 of those wrappers.

Strong

server.py server/tool_bundles/*_impl

The friction

server.py opens with "thin bootstrap orchestrator", then continues for ~3,200 lines registering 50 @mcp.tool wrappers. Each wrapper repeats the same five-step ritual: receive typed Pydantic params, elicit missing fields, dump Pydantic to dicts, dispatch to a sync _impl through asyncio.to_thread, parse the JSON string back to a dict.

Deletion test

Delete the wrappers and the elicitation/serialization rituals reappear in 50 different files, one per impl. The wrappers hold real glue. But the glue is the same glue repeated 50 times. That's the signature of a missing abstraction.

Candidate deepening

Define the wrapper ritual once. A register_tool_bundle(name, impl, ...) registration function takes the impl, a list of FieldSpecs, and metadata; it produces the @mcp.tool wrapper that handles elicitation, dumping, the thread bounce, and JSON parsing. Each tool becomes a 5-10 line registration block.

Candidate 2

ApiContext: collapse the 22-parameter impl signature

PM lead: every tool today drags 7 infrastructure plumbing arguments through every internal call. Bundling them as one context object simplifies tests and removes a documented technical-debt marker.

Strong

server/tool_bundles/crud.py (~1,800 lines)

The friction

create_complete_entity_impl has 22 positional parameters, starting with seven infrastructural ones before getting to a business field. The first thing the body does is build a cfg dict to thread these through helpers. A NOSONAR comment names the fix that hasn't been done.

Candidate 3

_helpers.py is a seven-concern junk drawer

PM lead: nineteen helpers serving seven unrelated concerns share one 1,298-line file, so every bug fix touches the same crowded file. Splitting it by concern makes each piece easy to find and easy to test.

Strong

server/tool_bundles/helpers.py

The friction

_helpers.py is 84KB / 49 top-level defs / 1,298 LOC across seven unrelated concerns. The documentation openly catalogues the spread, which is itself a tell.

Open questions

  • Should Candidate 2 stage 1 happen as a pure rename PR, or bundled with a behavior change? Bias: pure rename to keep blast radius small.
  • Does Candidate 1's register_tool_bundle need to support tools that already exist as classes, or only impls written for the new API?
  • What is the right migration window before Candidate 3's old _helpers.py shim can be deleted?

Top recommendation — sequenced plan

Lead with the narrow first stage of Candidate 2, then unfurl the rest.

The candidates compose. Doing them in the wrong order forces rework. Sequenced:

  1. Candidate 2 stage 1 — BundleCfg dataclass. Pure rename, lowest blast radius, hits two existing NOSONAR markers.
  2. Candidate 3 — split _helpers.py. Once BundleCfg is in, helpers stop needing 4-scalar threading everywhere.
  3. Candidate 2 stage 2 — full ApiContext. Promote BundleCfg to carry the full infra bundle.
  4. Candidate 1 — declarative tool registration. Now easy to write because impls take ApiContext.
