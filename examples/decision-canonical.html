Decision

In the context of three services duplicating Redis client setup, cooldown semantics, and test escape hatches, facing recurring bug-fix-replay cost and a duplicated infrastructure spend, we decided to extract a single shared LazyRedisCache module and have all three services compose it to achieve one place to fix bugs and a ~40% Redis cost reduction, accepting a one-sprint migration window and one new shared module to govern.

Context

Currently idempotency.py, reference.py, and session_cache.py each implement their own lazy Redis client with cooldown-based retry. The pattern is the same; the implementations have drifted. A bug fix shipped to one (cooldown jitter) had to be replayed in the others by hand, and one missed the fix for two weeks. The shared cache also reduces Redis connection count by ~3×.

Current vs proposed

AspectCurrent (three lazy clients)Proposed (one shared cache)
Bug fix surface3 files, replay required1 file, fix applies everywhere
Redis connections~12 per pod (3 clients × 4 pods)~4 per pod (shared)
Test substitutionEach test resets module globalsInject a fake; no globals to clear
Service ownershipEach service owns its cacheOne module owned by platform team

Consequences

  • Positive single place to fix Redis bugs; ~40% Redis cost reduction; cleaner test substitution via injection.
  • Neutral the shared module needs an owner (platform team); cache observability dashboards split into one shared view + per-service breakdowns.
  • Negative a bug in the shared cache breaks three services at once; ownership transitions during incidents need clearer escalation.

Reversibility

Mostly reversible If the shared cache proves to be wrong, each service can re-add its own lazy client by re-vendoring the file. The blocker would be data shape changes (cache keys we share that would diverge again); none planned. Decision can be revisited at the end of Q3 based on incident frequency.

  1. DR-001: Extract server/redis_cache.py as the shared lazy-cache module. Rationale: the three current callers (idempotency.py, reference.py, session_cache.py) all duplicate the cooldown + lazy-client pattern. Implementation note: create the module with LazyRedisCache.get/set/reset; migrate one caller; ship under feature flag; migrate the other two.

  2. DR-002: Preserve module-global escape hatches for tests during the one-sprint dual-running window. Rationale: existing tests reset module globals; keeping the escape hatches stops the migration from also rewriting 40+ tests at once.
