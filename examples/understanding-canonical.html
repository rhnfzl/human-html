What this thing is

The "request chain" is the sequence of services a user chat message passes through, from the browser to the Platform API and back. Four services participate, each owning a distinct responsibility: routing, agent orchestration, tool dispatch, business API.

How it works

The user types in the web UI. The Django frontend gateway authenticates the session and proxies the chat request to the Agent Service (FastAPI). The Agent Service runs the LLM agent loop, which calls tools on the MCP server. MCP translates each tool call into a Platform API request. Results flow back through the same chain.

flowchart LR U[Browser] --> FE[Front End] FE -->|SSO JWT| GW[Gateway Service
Django proxy] GW -->|forwards JWT| AGT[Agent Service
FastAPI agent loop] AGT -->|X-Acme-* headers| MCP[MCP Server
tool dispatch] MCP -->|basic auth| API[Platform API] API -.JSON.-> MCP MCP -.results.-> AGT AGT -.SSE chunks.-> GW GW -.SSE chunks.-> FE FE -.rendered chat.-> U

Gotchas

  • "Chat broken" tickets are usually Gateway Service: the proxy layer fails before the Agent Service ever sees the request, so Agent Service logs are empty when the user reports the issue.
  • Auth headers don't pass through transparently: the SSO JWT from the FE is unpacked at Gateway Service; Agent Service sends MCP a different shape (X-Acme-Token + X-Acme-Client-Id + X-Acme-Client-Secret) extracted from the JWT's service_tokens claim.
  • SSE buffering can hide errors: a network hiccup between GW and FE looks identical to AGT crashing mid-response. Check both before assuming one or the other.
Deep dive - JWT claim shape and how the extraction happens in each layer

Gateway Service receives the SSO JWT in the Authorization: Bearer <jwt> header. Inside the JWT payload, the service_tokens claim is an object:

{
  "sub": "user-1234",
  "service_tokens": {
    "api_token": "<opaque Platform bearer>",
    "client_id": "<tenant client id>",
    "client_secret": "<tenant secret>"
  },
  "iat": ..., "exp": ...
}

Gateway Service extracts service_tokens on each chat call and forwards the three values as the three X-Acme-* headers when calling Agent Service's /chat/completions endpoint. Agent Service then re-presents them to MCP, which proxies to the Platform API as basic auth. This is why a "missing client_id" error in MCP almost always originates in Gateway Service's JWT extraction, not in Agent Service.

import { jwtVerify, importSPKI } from "jose";

async function forwardToAgentService(req: Request): Promise<Response> {
  const token = req.headers.get("authorization")?.replace("Bearer ", "");
  const key = await importSPKI(SSO_PUBLIC_KEY_PEM, "RS256");
  const { payload } = await jwtVerify(token!, key);
  const ai = payload.service_tokens as { api_token: string; client_id: string; client_secret: string };
  return fetch(`${AGENT_SERVICE_URL}/chat/completions`, {
    method: "POST",
    headers: {
      "X-Acme-Token": ai.api_token,
      "X-Acme-Client-Id": ai.client_id,
      "X-Acme-Client-Secret": ai.client_secret,
      "Content-Type": "application/json",
    },
    body: await req.text(),
  });
}
# Bypass Gateway Service to probe Agent Service directly with the three headers.
# Useful when a chat ticket comes in and you want to isolate whether the bug is
# in GW's JWT extraction or downstream.

API_TOKEN=$(jq -r .service_tokens.api_token < jwt-payload.json)
CLIENT_ID=$(jq -r .service_tokens.client_id < jwt-payload.json)
CLIENT_SECRET=$(jq -r .service_tokens.client_secret < jwt-payload.json)

  -H "X-Acme-Token: $API_TOKEN" \
  -H "X-Acme-Client-Id: $CLIENT_ID" \
  -H "X-Acme-Client-Secret: $CLIENT_SECRET" \
  -H "Content-Type: application/json" \
