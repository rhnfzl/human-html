# Author a human-readable HTML artifact

When the user wants the next deliverable to land as an HTML page a teammate can actually read (instead of a long Markdown wall that gets skimmed), this command picks the right kind, scaffolds the file under `docs/human-html/`, and populates it. Markdown stays for agent scratch, durable references, ticket notes, drafts, and meetings. HTML is for human review surfaces.

## When to use

- The user typed `/human-html` and wants a new artifact (plan, review, architecture explainer, understanding doc, research synthesis, decision aid, prototype, status report, or incident postmortem).
- The output of the current task is going to a teammate, a tech lead, or a PM who has ten minutes to read it and make a decision.
- The user wants to update an existing artifact under `docs/human-html/` (revise the body, add a chart, refresh metrics).
- A grilling session, a retrospective, or a brainstorm produced a locked plan that needs a human-facing artifact.

## When NOT to use

- The output is an agent-to-agent handoff (Markdown parses better for that).
- The output is a source-of-truth spec that gets edited weekly in git (Markdown wins for git review; HTML diffs are noisy).
- The output is under ~20 lines (a chat reply or a short brief is enough).
- The artifact is not intended for a human reader.
- The workspace has not adopted the human-html harness yet (run `init` first; see the workflow).

## Your job

You scaffold the artifact with the right filename and metadata, populate the body, then leave the gallery `index.html` auto-regeneration to the harness hook.

## Kinds

The harness recognises nine kinds. Pick the one that matches the artifact:

- `plan`: forward-looking work plan, includes scope, milestones, risks.
- `review`: critique of a code change, PR, design, or system.
- `architecture`: explainer of how a system or feature works.
- `understanding`: what the team learned by exploring a codebase or system.
- `research`: synthesis of external evidence (web search, papers, blog posts).
- `decision`: aid that locks a choice with options and rationale.
- `prototype`: live demo or experimental UI.
- `status`: snapshot of where work stands (sprint review, audit report).
- `incident`: postmortem with a timeline, impact, root cause, and corrective actions.

## Workflow

1. **Confirm the kind.** Ask the user which of the nine kinds fits. If unclear from context, recommend one based on the task.

2. **Confirm the slug.** Format: short, kebab-case, descriptive. Examples: `auth-jwt-rollout`, `cache-topology-decision`, `q2-2026-payments-status`. Avoid a `team` prefix.

3. **Check the harness is initialized.** From the workspace root:
   ```bash
   ls docs/human-html/
   ```
   If the directory does not exist, run:
   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/human_html_artifacts.py init
   ```
   This seeds `docs/human-html/` with the README (contract restatement), `GLOSSARY.md`, and an empty `index.html`.

4. **Scaffold the artifact.** Run:
   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/human_html_artifacts.py new <kind> "<title>"
   ```
   The script writes `docs/human-html/YYYY-MM-DD-<kind>-<slug>.html` with required metadata (artifact-kind, artifact-audience=human, artifact-created, artifact-source, data-human-html-artifact=true) and refreshes `index.html`.

5. **Populate the body.** Open the new file and replace the scaffold body. Lead with plain-language context (what this gives the reader, why it matters), then go into technical detail. Use semantic HTML: `<h2>`, `<h3>`, `<ul>`, `<table>`, `<pre><code>`, `<img>`. Inline a diagram (SVG or Mermaid-rendered SVG) where the artifact is architectural. Inline a chart where the artifact has metrics. Color-code risk callouts.

6. **Honor branding** if the workspace has a brand profile (fonts, accent colors). Reference the project's existing artifacts under `docs/human-html/` for the in-use style.

7. **Validate.** Run:
   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/human_html_artifacts.py check
   ```
   The check enforces filename format, metadata presence, valid local links, the no-root-HTML rule, and the content contract.

8. **Hand off.** Tell the user the path. The autoindex hook will refresh `index.html` on the next tool call; if you want it now, run the `index` subcommand manually.

## Output

The user sees:

- The kind and slug confirmed.
- The full path of the new artifact (e.g., `docs/human-html/2026-07-08-decision-cache-topology.html`).
- A clean `check` output.
- A reminder that `index.html` is auto-regenerated by the autoindex hook; no need to edit it by hand.

## Updating an existing artifact

If the user wants to revise an existing artifact:

1. Find the file under `docs/human-html/`. Honor the existing filename; do not rename (the date in the filename is the creation date, not the last-edit date).
2. Edit the body, preserving the metadata block in `<head>`.
3. Run `check` after editing.
4. Optionally bump the `artifact-source` to note the revision pass.

## Reference

The canonical skill (script, hooks, baseline allowlist, per-workspace customization knob, full content contract) lives at `"$CLAUDE_PLUGIN_ROOT"/SKILL.md`. The pattern comes from the industry layer-split framing that converged in mid-2026: HTML for human review, Markdown for agent memory.
